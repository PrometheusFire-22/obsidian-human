# OpenClaw Latency Root Cause — Apr 5

## Finding

Cornelius was sending 245k-248k tokens upstream on every single Telegram message, causing 3-minute response times.

The cause was NOT the workspace files (those total ~51KB / ~13k tokens).

**Root cause: accumulated session history.**
Active session file: `~/.openclaw/agents/main/sessions/005b39b7.jsonl`
- Size: 1.6MB, 501 turns of conversation history
- OpenClaw injects the full session history into every LLM call
- 501 turns * average message size = ~245k tokens per request

## Fix Applied (CLA-77)

Added to `~/.openclaw/openclaw.json`:

1. **contextPruning** (cache-ttl mode) — trims old tool outputs from in-memory context before each LLM call. Tool results older than 5m get softTrimmed or hardCleared. This alone reduces token bloat significantly.

2. **session.reset** (daily at 4am) — starts a fresh session every day. Prevents unbounded history growth. Also resets after 2h idle.

3. **session.maintenance** — enforces pruning of sessions older than 30d, max 500 entries, rotates at 10mb.

## Still needed

- Type `/compact` in Telegram to compress the current 501-turn session immediately
- Monitor token count in Control UI after compact — should drop below 30k
- CLA-77 stays open until response times are confirmed under 15s

## Lesson

The memory SQLite DB (8.4MB) is NOT the problem — it only has 2 chunks. Session history is the token budget killer. Always check session file size when diagnosing latency
